# Issues and Resolutions

This document records bugs, technical issues, and architectural challenges encountered during the development of the Breakfast Minecraft Bedrock addon, along with their resolutions.

---

## Issue 1: Block Interactions (Right-Clicking) Not Firing Script Events

### Symptom
The Butcher Block and Griddle custom blocks were craftable, placeable in the world, and mineable/recoverable. However, right-clicking them with items (like the knife or cookable ingredients) did not place items or trigger any scripting logic in `main.js`. The player's arm would swing, but no action bar text, sound effects, or block state changes occurred.

### Investigation
1. Checked script registration: The scripts in `main.js` registered custom components `"breakfast:butcher_block_component"` and `"breakfast:griddle_component"` under `system.beforeEvents.startup.subscribe`.
2. Checked block definitions: The block JSON files referenced these components inside `"components"` under `"minecraft:custom_components"`.
3. Analyzed API/Engine support: In Minecraft Bedrock, stable custom block components (`minecraft:custom_components`) were introduced and standardized starting in engine version `1.21.10`.
4. Found mismatch: The block JSON files had `"format_version": "1.21.0"`, and the behavior/resource pack manifests specified `"min_engine_version": [1, 21, 0]`. Because the targeted versions were below the threshold of `1.21.10` (when custom components were released from experimental), the game engine discarded/ignored the `"minecraft:custom_components"` component in the JSON definitions.

### Resolution
* Updated `"format_version"` in [butcher_block.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/butcher_block.json) and [griddle.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/griddle.json) to `"1.21.20"`.
* Updated `"min_engine_version"` in the Behavior Pack [manifest.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/manifest.json) and Resource Pack [manifest.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_RP/manifest.json) to `[1, 21, 20]`.
* Repackaged the addon using the python build tool.

Following these changes, the game engine correctly processes the `minecraft:custom_components` array, binding the custom blocks to their respective JavaScript handlers and enabling right-click interactions.

## Issue 2: Pack Import Failure ("Failed to import - not a valid zip archive")

### Symptom
When attempting to import the newly packaged `Breakfast_1.0.4.mcaddon` file, Minecraft Bedrock Edition failed to import the pack, displaying the error message: `"Failed to import - not a valid zip archive"`.

### Investigation
1. Inspected zip entries: Ran a PowerShell command using `.NET`'s `ZipFile` class to list the entries of the generated `.mcaddon` file.
2. Found path separators mismatch: The entries generated by PowerShell's native `Compress-Archive` cmdlet used Windows backslashes (`\`) for file path separators (e.g. `Breakfast_BP\manifest.json`).
3. Checked Minecraft Bedrock requirements: Minecraft's internal zip extraction utility strictly requires standard forward slashes (`/`) for directory separators inside the archive. Because the paths contained backslashes, Minecraft rejected the archive as corrupt or invalid.

### Resolution
* Rewrote [pack_addon.ps1](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/tools/pack_addon.ps1) to use the .NET `[System.IO.Compression.ZipArchive]` class directly.
* Normalised all directory separators inside the script by substituting backslashes (`\`) with forward slashes (`/`) before adding them to the zip package.
* Re-ran the build script to output the corrected `Breakfast_1.0.4.mcaddon` file with standardized path separators.

## Issue 3: Right-Clicking Block Interactions Fail Silently

### Symptom
Even after registering block components and correcting block JSONs, players were still unable to right-click items onto the Butcher Block or Griddle. The arm swung, but the action bar and placement logic did not execute. No errors were outputted.

### Investigation
1. Analyzed `main.js`: Interaction functions retrieve the player's inventory using `player.getComponent("inventory")`.
2. Checked `@minecraft/server` API specifications: The parameter for `getComponent()` must be namespace-prefixed. The component is `"minecraft:inventory"`, not `"inventory"`. 
3. Identified silent failure: Because `"inventory"` was used, the call returned `undefined`. The validation check `if (!inventory || !inventory.container) return;` caught this and exited the handler early without throwing a script exception, hiding the bug.

### Resolution
* Replaced all instances of `player.getComponent("inventory")` with `player.getComponent("minecraft:inventory")` in [main.js](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/scripts/main.js).

## Issue 5: Blocks Uncraftable due to Invalid Format Version

### Symptom
Following the target version bump, the custom blocks (Butcher Block and Griddle) became completely uncraftable. 

### Investigation
1. Analyzed recipe JSON files: Crafting recipes are defined with standard result item IDs (`breakfast:butcher_block` and `breakfast:griddle`). The recipes themselves had no issues.
2. Checked block parsing: In Bedrock Edition, if a custom block definition fails to load or parse due to an error, the block item is unregistered. This automatically disables any recipes producing that block.
3. Identified format version mismatch: We had set `"format_version"` to `"1.26.20"` in the block JSON files. However, Bedrock block JSON format schemas are incremented only on major/minor versions (e.g. `1.26.0`, `1.26.10`). Because `"1.26.20"` is not a recognized format version schema, the block parser rejected both files completely.

### Resolution
* Updated `"format_version"` to `"1.26.0"` in both [butcher_block.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/butcher_block.json) and [griddle.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/griddle.json).
* Kept `min_engine_version` as `[1, 26, 20]` in the manifests to ensure the overall pack still correctly targets the `26.2` release.

## Issue 6: Griddle Interaction Fails (Custom Components V2 Inline Format)

### Symptom
Even though the Griddle block was placeable and registered, players were unable to place items onto it for cooking.

### Investigation
In Minecraft Bedrock 1.21.80+ / 1.26.x (such as 26.2), the original `"minecraft:custom_components"` array wrapper in block JSONs was deprecated and removed. All custom components must now be declared **inline** directly inside the block's `"components"` object (e.g. `"breakfast:griddle_component": {}`). Because we had declared them using the old array wrapper, the game client discarded the script linkage.

### Resolution
* Modified [butcher_block.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/butcher_block.json) and [griddle.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/blocks/griddle.json) to declare components directly inline (Custom Components V2 syntax).

## Issue 7: Butcher Block Recipe Does Not Appear (Legacy Plank ID)

### Symptom
The Butcher Block recipe did not show up in the crafting table and was uncraftable.

### Investigation
1. Analyzed [butcher_block_craft.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/recipes/butcher_block_craft.json): The recipe linked the `P` character using `"item": "minecraft:planks"`.
2. Identified deprecated ID: In modern Bedrock Edition (1.20.0+), `"minecraft:planks"` is no longer a valid item ID (since planks are named `minecraft:oak_planks`, etc.). The tag representing planks is `"minecraft:planks"`, which must be referenced as `"tag": "minecraft:planks"` in recipe keys. Because the recipe used `"item"` instead of `"tag"`, the recipe parser rejected it.

### Resolution
* Changed `"item": "minecraft:planks"` to `"tag": "minecraft:planks"` in [butcher_block_craft.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/recipes/butcher_block_craft.json) and [griddle_craft.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/recipes/griddle_craft.json).

## Issue 8: Custom Foods Inedible

### Symptom
Custom foods like Miner's Skillet or Bacon given via the creative menu could not be consumed.

### Investigation
In Bedrock Edition item JSON files, simply having `"minecraft:food"` is not enough to make an item edible; it also requires `"minecraft:use_animation": "eat"` and `"minecraft:use_modifiers"` specifying the `use_duration` (standard is 1.6s). Because these were missing, the player was unable to perform the use action/eat animation when holding the items.

### Resolution
* Created and executed a Node.js utility script [update_items.js](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/tools/update_items.js) that added `"minecraft:use_animation": "eat"` and `"minecraft:use_modifiers"` (with `use_duration` set to `1.6` for all foods, and `0.8` for `fried_egg.json`) to all 19 custom food item files.

## Issue 9: Custom Blocks Still Uncraftable (Low Recipe Format Version)

### Symptom
The Butcher Block and Griddle remained completely uncraftable. 

### Investigation
Even though we updated the ingredient keys from `"item": "minecraft:planks"` to `"tag": "minecraft:planks"`, the recipe files still targeted `"format_version": "1.12.0"`. The `1.12.0` recipe parser does not support the `tag` property inside recipe keys. As a result, the recipe files failed validation and were discarded by the game engine, making the blocks uncraftable.

### Resolution
* Updated `"format_version"` to `"1.21.0"` in [butcher_block_craft.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/recipes/butcher_block_craft.json) and [griddle_craft.json](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/Breakfast_BP/recipes/griddle_craft.json).

## Issue 10: Custom Foods Edible When Full

### Symptom
Players in Survival mode were able to eat custom food items (like bacon or miner's skillet) even when their hunger bar was fully replenished.

### Investigation
The `minecraft:food` item component has an optional boolean parameter `can_always_eat` (which determines if the item is edible when full). In certain Bedrock updates, custom food items default this parameter to `true` if it is omitted.

### Resolution
* Updated the Node.js helper utility script [update_items.js](file:///c:/Users/brett/Code/2026/Antigravity/Breakfast/tools/update_items.js) to explicitly add `"can_always_eat": false` within the `"minecraft:food"` component of all 19 custom food item definitions. 
* Re-ran the utility script to apply the changes across the behaviour pack items.
